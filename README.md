## 氣泡排序法 (Bubble Sort)

### 簡述原理
重複比較兩兩相鄰的元素，如果順序錯誤就交換，將大的元素慢慢搬到陣列末尾。

### 複雜度分析
- 平均時間複雜度: O(n²)
- 最壞時間複雜度: O(n²)
- 最好時間複雜度: O(n) (當陣列已排序時)
- 空間複雜度: O(1)

### 模擬實作內容
氣泡排序在數據量增大時，執行時間呈現劇烈的二次方增長。

## 選擇排序法 (Selection Sort)

### 簡述原理
在陣列中找到最小的元素，跟第一個元素交換；然後在剩下的元素中再找最小的，跟第二個交換，依此類推。

### 複雜度分析
- 平均時間複雜度: O(n²)
- 最壞時間複雜度: O(n²)
- 最好時間複雜度: O(n²)
- 空間複雜度: O(1)

### 模擬實作內容
選擇排序的特性是無論數據狀態如何，其比較次數固定，因此三種情況耗時接近。

## 插入排序法 (Insertion Sort)

### 簡述原理
像整理撲克牌一樣，將元素一個一個插入到已經排序好的序列中。

### 複雜度分析
- 平均時間複雜度: O(n²)
- 最壞時間複雜度: O(n²)
- 最好時間複雜度: O(n) (當陣列已接近排序時)
- 空間複雜度: O(1)

### 模擬實作內容
插入排序在「已排序」狀態下具有極佳效能，但在隨機與倒序情況下仍屬二次方成長。

## 合併排序法 (Merge Sort)

### 簡述原理
採用分而治之法，先遞迴地將陣列分割成兩半，排序完成後再合併起來。

### 複雜度分析
- 平均時間複雜度: O(n log n)
- 最壞時間複雜度: O(n log n)
- 最好時間複雜度: O(n log n)
- 空間複雜度: O(n)

### 模擬實作內容
合併排序表現非常穩定，數據量增加十倍時，耗時僅增加約十餘倍，而非百倍。

## 快速排序法 (Quick Sort)

### 簡述原理
選擇一個元素作為基準 (pivot)，將陣列分成大於和和小於基準的兩部分，再遞迴地對兩部分進行排序。

### 複雜度分析
- 平均時間複雜度: O(n log n)
- 最壞時間複雜度: O(n²) (當基準選擇極不平衡時)
- 最好時間複雜度: O(n log n)
- 空間複雜度: O(log n) (由於遞迴堆疊)

### 模擬實作內容
快速排序在大多數情況下是實測最快的，但在倒序且選取末端為基準時會明顯變慢。

## 綜合比較
綜合比較來看, 氣泡、選擇和插入排序法在平均和最壞情況下都是 O(n²) 的時間複雜度, 效率較低, 適合小型資料,優點是空間複雜度低, 且氣泡和插入排序法是穩定的。

合併和快速排序法則速度較快, 平均可達 O(n log n), 適合大型資料, 但合併排序法需要額外空間, 快速排序法在最壞情況下會退化, 且通常是不穩定的。

## 學習心得
這次排序演算法的實作實驗讓我對「時間複雜度」有了具象化的認知。原本在課堂上聽老師講 $O(n^2)$ 和 $O(n \log n)$ 的差別時，總覺得只是數學上的抽象代號，但當我親自將數據量從一千筆拉升到十萬筆時，那種效能上的斷崖式差距真的讓我感到震懾。看到氣泡排序在處理十萬筆隨機數據時，電腦需要運算超過二十秒才能完成，而快速排序卻能在不到二十毫秒內就瞬間排好，我才真正體會到演算法優劣對於程式效能的影響是多麼巨大。在實作過程中，我也觀察到不同演算法在特定場景下的「求生本能」。像是插入排序雖然在數據量大時表現普通，但當我測試「已排序」的數據時，它展現出的驚人效率甚至超越了複雜的快速排序，這點讓我學到沒有絕對最強的演算法，只有最適合場景的選擇。另外，合併排序那種無論數據如何分佈都始終如一的穩定性，以及快速排序在處理倒序數據時可能的效能退化，都讓我意識到在開發系統時，必須考量到最壞情況發生的可能性。這次的視覺化網頁製作也讓我收穫頗豐。透過 JavaScript 動畫將抽象的比較過程轉化為跳動的長條圖，不僅讓程式碼除錯變得直觀，也讓我在編寫遞迴函數時，腦袋裡能有一個清晰的物理圖像。從最初的單一選單到後來調整成明亮華麗的毛玻璃質感，這個過程不僅是技術上的磨練，更是一次結合邏輯思維與視覺美學的實踐，讓我對 EECS 領域的實務開發有了更深一層的興趣。
